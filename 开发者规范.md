# React 商城系统开发者规范

本文档面向两类使用者：

- 人类开发者：作为本项目的编码规范、页面验收标准和提交前检查清单。
- 大模型助手：作为接手任务时必须遵守的工程 Prompt，确保在完成 `README.md` 要求的同时尽量保证代码质量、视觉一致性和可维护性。

本项目目标是完成一套 React 商城系统，包含前台产品端与后台管理端。所有开发都必须优先满足 `README.md` 的作业要求，再追求加分项、视觉优化和代码重构。

## 1. 项目定位

### 1.1 产品目标

构建一个完整的像素主题商城系统，用户可以浏览商品、搜索商品、查看分类、加入购物车、创建订单、模拟支付、查看订单；后台管理人员可以登录后台，并基于角色权限进入对应管理模块，至少完成商品管理、分类管理、订单管理中的一个完整 CRUD 管理功能。

### 1.2 技术栈

当前项目使用：

| 类型 | 技术 |
|------|------|
| 构建工具 | Vite |
| 前端框架 | React 19 |
| 路由 | React Router 7 |
| 状态管理 | React Context、服务层、本地状态 |
| 数据模拟 | localStorage、Mock Service |
| 样式 | 原生 CSS 分层样式系统 |
| 代码检查 | ESLint |
| 提交检查 | `npm run lint`、`npm run build`、`npm run check`、`npm run pack` |

### 1.3 视觉方向

视觉 thesis：

像素主题商城应该呈现粉白、奶油、玫瑰、金色点缀的可爱像素感，同时保留商城系统应有的清晰导航、商品信息密度和交易流程可信度。

内容 thesis：

前台围绕浏览、筛选、购买、订单追踪组织；后台围绕登录、权限、商品或订单管理组织；每个页面必须有明确的任务，不做单纯展示页。

交互 thesis：

交互应该轻快但克制。优先使用清晰的 hover、选中、禁用、加载、空状态、表单校验反馈和支付流程状态变化。动效只用于强化层级、状态变化或操作反馈。

## 2. README 作业要求转化为验收清单

开发必须先保证以下内容完成。任何视觉优化、抽象封装、性能优化都不能替代功能完成度。

### 2.1 前台功能必须完成

| 模块 | 必须能力 | 验收标准 |
|------|----------|----------|
| 商城主页面 | 搜索框、轮播图、热门商品展示 | 首页能看到搜索入口、轮播区域、热门商品列表；商品可进入详情 |
| 分类页 | 展示所有商品分类索引 | 分类可点击或筛选，能看到分类下商品 |
| 购物车 | 查看商品、修改数量、删除、选中结算 | 数量变化影响总价；删除后刷新不恢复；未选中商品不参与结算 |
| 我的页面 | 展示用户信息、订单入口 | 登录后显示用户信息与订单列表入口；未登录引导登录 |
| 商品详情页 | 展示商品信息，加入购物车或立即购买 | 能加入购物车；立即购买能进入创建订单 |
| 创建订单页 | 确认地址、商品清单、生成订单 | 订单生成后能跳转支付页或订单详情 |
| 支付页面 | 模拟支付流程 | 支持支付成功或失败状态；成功后订单状态更新 |
| 订单详情页 | 查看订单状态、商品信息、物流信息 | 订单详情和列表数据一致 |

### 2.2 后台功能必须完成

| 模块 | 必须能力 | 验收标准 |
|------|----------|----------|
| 后台登录 | 独立后台登录页 | 未登录不能进入后台管理页 |
| 权限管理 | 角色访问控制 | 管理员和普通运营看到的模块不同 |
| 管理模块 | 商品、分类、订单三选一完整 CRUD | 至少一个模块支持新增、查看、编辑、删除或上下架，并与前台数据联动 |

### 2.3 通用功能必须完成

| 功能 | 必须能力 | 验收标准 |
|------|----------|----------|
| 用户登录 | 前台登录或注册 | 未登录不能购物车结算和下单 |
| 数据联动 | 前后台数据一致 | 后台修改商品后，前台商品展示同步变化 |
| 数据持久化 | localStorage 或 Mock 持久化 | 刷新页面后商品、购物车、登录状态、订单不无故丢失 |

### 2.4 加分项优先级

推荐优先顺序：

1. 数据持久化。
2. 表单验证。
3. 响应式布局。
4. 商品或订单分页。
5. 支付倒计时或二维码模拟。
6. 性能优化。
7. 单元测试。
8. 部署上线。

## 3. 目录职责规范

当前项目目录必须保持清晰分层。新增文件必须放在对应目录，不允许为了方便把所有逻辑堆进 `App.jsx` 或单个页面文件。

```text
src/
  main.jsx                 React 应用启动入口
  App.jsx                  应用壳，承载 <Outlet />、全局布局或全局弹窗
  index.css                应用级 CSS 入口，具体规则以样式指南为准
  App.css                  App 组件专属样式，具体规则以样式指南为准
  router.jsx               当前路由入口
  router/                  前台和后台路由拆分配置
  pages/                   页面级组件
  components/              可复用 UI 或业务组件
  contexts/                全局 Context Provider
  services/                业务服务层，封装商品、订单、用户、购物车等数据操作
  api/                     接口封装或模拟 API
  store/                   全局状态或类型化 store
  hooks/                   自定义 Hooks
  utils/                   工具函数
  mock/                    初始 Mock 数据
  assets/                  图片、图标、静态素材
  styles/                  样式系统
```

### 3.1 页面目录

`src/pages/` 存放能被路由直接渲染的页面组件。

命名规则：

| 页面类型 | 文件名示例 |
|----------|------------|
| 前台首页 | `HomePage.jsx` |
| 分类页 | `CategoryPage.jsx` |
| 购物车 | `CartPage.jsx` |
| 我的页面 | `ProfilePage.jsx` |
| 商品详情 | `DetailPage.jsx` 或 `ProductDetailPage.jsx` |
| 创建订单 | `CreateOrderPage.jsx` |
| 支付 | `PayPage.jsx` |
| 订单列表 | `OrderListPage.jsx` |
| 订单详情 | `OrderDetailPage.jsx` |
| 后台登录 | `AdminLoginPage.jsx` |
| 后台商品管理 | `AdminProductsPage.jsx` |
| 后台订单管理 | `AdminOrdersPage.jsx` |
| 后台权限管理 | `AdminRolesPage.jsx` |

页面组件只负责：

- 组织页面布局。
- 调用服务层或 Context 获取数据。
- 管理页面级状态。
- 处理页面级 loading、empty、error。
- 组合公共组件。

页面组件不应该：

- 直接散落大量 localStorage 操作。
- 写重复的商品卡、订单卡、按钮样式。
- 混入后台和前台无关逻辑。
- 承担复杂数据清洗和持久化细节。

### 3.2 组件目录

`src/components/` 存放可复用组件，建议按场景分组：

```text
components/
  common/          Button、Modal、EmptyState、PageHeader 等
  h5/              前台商城复用组件
  admin/           后台管理复用组件
```

组件设计规则：

- 组件必须有单一职责。
- 组件名使用 PascalCase，例如 `ProductCard`。
- 组件文件名与组件名一致，例如 `ProductCard.jsx`。
- 事件回调使用 `onXxx` 命名，例如 `onAddToCart`、`onQuantityChange`。
- 布尔属性使用 `is`、`has`、`can` 前缀，例如 `isSelected`、`canEdit`。
- 组件内部不要偷偷读写全局状态，除非它本身就是容器组件。
- 可复用展示组件优先通过 props 接收数据。

### 3.3 服务层目录

`src/services/` 用于封装业务操作。组件调用服务层，而不是直接处理存储细节。

推荐服务：

| 服务 | 职责 |
|------|------|
| `goodService.js` | 商品列表、商品详情、搜索、分类、上下架 |
| `cartService.js` | 购物车读取、添加、数量修改、删除、勾选 |
| `orderService.js` | 创建订单、支付状态、订单列表、订单详情、发货 |
| `userService.js` | 前台登录、注册、退出、当前用户 |
| `adminService.js` | 后台登录、角色、权限 |

服务层函数必须：

- 函数名表达业务意图，例如 `getProductById`、`updateCartQuantity`。
- 返回稳定的数据结构。
- 对找不到数据、非法参数、重复提交做防御。
- 不在函数内部操作 DOM。
- 不依赖页面组件状态。

### 3.4 工具函数目录

`src/utils/` 只放无业务副作用的通用函数，例如：

- `formatPrice`
- `formatDate`
- `safeJsonParse`
- `generateId`
- `storage`

工具函数必须可测试、可复用、无 React 依赖。

## 4. React 开发规范

### 4.1 组件基本规则

必须使用函数组件和 Hooks。

推荐结构：

```jsx
function ProductCard({ product, onAddToCart }) {
  const isSoldOut = product.stock <= 0;

  return (
    <article className="pm-product-card">
      {/* JSX */}
    </article>
  );
}

export default ProductCard;
```

规则：

- 每个文件默认只导出一个主组件。
- 组件名必须大写。
- 不在 render 中修改状态、写 localStorage、发请求。
- 不把复杂表达式直接塞进 JSX，先用变量命名。
- 条件渲染必须考虑 loading、empty、error、unauthorized。
- 列表渲染必须使用稳定 key，优先使用业务 id，不能使用随机数。

### 4.2 Hooks 使用规范

必须遵守 React Hooks 规则：

- Hooks 只能在组件顶层或自定义 Hook 顶层调用。
- 不在 if、for、回调函数里调用 Hook。
- `useEffect` 只处理副作用，不用它计算可以在 render 阶段得到的派生数据。
- 依赖数组必须完整，不能为了消除警告随意删依赖。
- 组件卸载时需要清理定时器、事件监听、异步订阅。

`useEffect` 推荐用途：

- 首次加载数据。
- 同步 document title。
- 订阅和取消订阅。
- 定时器。
- localStorage 与 React 状态同步。

不推荐用途：

- 把 props 复制进 state。
- 计算筛选后的数组。
- 根据另一个 state 立刻 setState。

### 4.3 React Compiler 相关规范

项目已使用 React Compiler preset。默认不要滥用 `React.memo`、`useMemo`、`useCallback`。

只有这些场景才考虑手动 memo：

- 子组件依赖引用稳定性，并且有实际性能问题。
- 列表项很多，重复渲染明显影响交互。
- 某个计算确实昂贵，并且输入依赖明确。
- 第三方组件要求稳定回调引用。

禁止为了“看起来高级”到处包 `useMemo`、`useCallback`。

### 4.4 状态管理分层

优先按以下顺序选择状态位置：

| 状态类型 | 推荐位置 |
|----------|----------|
| 输入框当前值、弹窗开关 | 当前组件 |
| 页面筛选条件、分页参数 | 页面组件 |
| 当前用户、购物车数量、主题 | Context |
| 商品、订单、角色等业务数据 | service + localStorage |
| 可从已有数据计算出的值 | 不存 state，直接计算 |

不要把所有状态都放进 Context。Context 只用于真正跨页面共享的数据。

### 4.5 表单规范

所有登录、注册、创建商品、编辑商品、地址、支付确认等表单必须有基础校验。

最低要求：

- 必填项不能为空。
- 手机号、邮箱、价格、库存等字段格式正确。
- 数量不能小于 1。
- 价格不能小于 0。
- 提交失败时给出错误提示。
- 提交成功时给出明确反馈或跳转。
- 按钮在提交中应禁用，避免重复提交。

错误提示应靠近对应字段，不只用 `alert`。

### 4.6 路由规范

当前项目使用 React Router。推荐使用一个根路由承载 `App`，在 `App` 内用 `<Outlet />` 渲染子路由。

推荐路由规划：

| 路径 | 页面 |
|------|------|
| `/` | 前台首页 |
| `/home` | 前台首页，可作为兼容路径 |
| `/category` | 分类页 |
| `/cart` | 购物车 |
| `/profile` | 我的页面 |
| `/login` | 前台登录注册 |
| `/detail/:goodId` | 商品详情 |
| `/createOrder/:goodId` | 创建订单 |
| `/pay/:orderId` | 支付页 |
| `/orderList` | 订单列表 |
| `/orderDetail/:orderId` | 订单详情 |
| `/admin/login` | 后台登录 |
| `/admin` | 后台首页或重定向 |
| `/admin/products` | 商品管理 |
| `/admin/orders` | 订单管理 |
| `/admin/roles` | 权限管理 |

路由规则：

- 前台和后台路由必须在 URL 上清晰区分。
- 需要登录的页面必须做登录保护。
- 后台页面必须做管理员登录保护。
- 后台管理模块必须做角色权限判断。
- 找不到页面时应提供 Not Found 或重定向，不要白屏。
- 页面跳转后数据必须和 URL 参数一致。

### 4.7 权限规范

至少定义两类后台角色：

| 角色 | 权限 |
|------|------|
| `admin` | 可访问商品、订单、权限等管理模块 |
| `operator` | 可查看订单或指定模块，不可访问权限管理 |

权限判断必须基于角色，不要只基于用户名字符串散落在页面里。

推荐权限结构：

```js
const rolePermissions = {
  admin: ['product:manage', 'order:manage', 'role:manage'],
  operator: ['order:view'],
};
```

页面和菜单都要基于权限渲染。没有权限时，不能只隐藏菜单，还要拦截直接输入 URL 的访问。

## 5. 数据与 localStorage 规范

### 5.1 数据来源

本作业允许用 localStorage 或 Mock 数据模拟后端。项目中建议采用：

- `mock/` 存初始数据。
- `services/` 封装读取、写入、更新、删除。
- `contexts/` 提供当前用户、购物车摘要、服务对象等全局能力。

### 5.2 localStorage key 命名

统一使用项目前缀，避免污染浏览器存储：

```text
pixelMall:products
pixelMall:categories
pixelMall:cart
pixelMall:orders
pixelMall:currentUser
pixelMall:adminUser
pixelMall:roles
pixelMall:theme
```

禁止使用含义不清的 key，例如 `data`、`list`、`user1`。

### 5.3 数据结构最低标准

商品：

```js
{
  id: 'p_001',
  name: '像素草莓蛋糕',
  categoryId: 'dessert',
  price: 39.9,
  originalPrice: 49.9,
  stock: 99,
  sales: 120,
  cover: '/path/to/image.png',
  images: [],
  description: '商品描述',
  status: 'onSale'
}
```

购物车项：

```js
{
  id: 'cart_001',
  productId: 'p_001',
  quantity: 2,
  selected: true
}
```

订单：

```js
{
  id: 'o_001',
  userId: 'u_001',
  items: [],
  address: {},
  totalAmount: 79.8,
  status: 'pendingPayment',
  logistics: [],
  createdAt: '2026-06-03T12:00:00.000Z'
}
```

订单状态推荐：

| 状态 | 含义 |
|------|------|
| `pendingPayment` | 待支付 |
| `paid` | 已支付 |
| `shipped` | 已发货 |
| `completed` | 已完成 |
| `cancelled` | 已取消 |
| `failed` | 支付失败 |

### 5.4 数据一致性

必须保证：

- 后台下架商品后，前台不应继续作为可购买商品展示。
- 后台修改商品名称、价格、库存后，前台详情和列表同步。
- 用户支付成功后，订单列表和订单详情状态同步。
- 购物车商品数量不能超过库存。
- 删除商品时，要处理购物车或订单中引用该商品的显示问题。

## 6. 样式系统规范

本规范不另写一套 CSS 规则。所有 CSS 入口、目录分层、导入顺序、token、主题、class 命名、组件样式、页面样式、响应式、状态类和像素主题视觉要求，统一以 `src/styles/docs/PIXEL_MALL_STYLE_GUIDE_PROMPT.md` 为唯一标准。

开发者和大模型在新增或修改任何样式前，必须先阅读并遵守：

```text
src/styles/docs/PIXEL_MALL_STYLE_GUIDE_PROMPT.md
```

本文只保留与开发流程相关的约束：

- 不在 `开发者规范.md` 中重复定义第二套 CSS 架构。
- 不新增绕过 `src/styles/index.css` 的页面级散乱引入。
- 不新增与样式指南冲突的颜色、命名、目录、导入顺序或视觉规则。
- 样式指南没有覆盖的问题，先沿用 `src/styles/` 已有文件和命名方式；确实需要新增规则时，优先更新 `PIXEL_MALL_STYLE_GUIDE_PROMPT.md`。
- 页面功能实现标准见本文第 7 节，具体视觉落地必须回到样式指南执行。

## 7. 页面实现标准

### 7.1 首页

必须包含：

- 顶部导航或移动端底部导航。
- 搜索框。
- 轮播或主视觉商品区域。
- 热门商品。
- 分类入口。
- 登录、购物车、订单或我的入口。

最低交互：

- 搜索能过滤商品或跳转结果页。
- 点击商品进入详情。
- 点击分类进入分类页或筛选商品。

### 7.2 分类页

必须包含：

- 分类列表。
- 当前分类选中状态。
- 分类下商品列表。
- 空分类提示。

### 7.3 购物车

必须包含：

- 商品勾选。
- 全选。
- 数量加减。
- 删除。
- 小计和总计。
- 去结算。

业务规则：

- 未登录访问购物车时引导登录。
- 数量不能小于 1。
- 数量不能超过库存。
- 已下架商品不能结算。
- 结算只包含选中商品。

### 7.4 商品详情页

必须包含：

- 商品图。
- 商品名称。
- 价格。
- 库存或状态。
- 详情描述。
- 加入购物车。
- 立即购买。

业务规则：

- 未登录点击加入购物车或立即购买时引导登录。
- 已下架或库存为 0 时禁用购买。
- URL 参数找不到商品时显示错误或空状态。

### 7.5 创建订单页

必须包含：

- 收货地址。
- 商品清单。

## 8. 大模型协作 Prompt

给大模型派发任务时，必须把本节作为系统级开发规范使用。

### 8.1 接手任务前必须读取

大模型在写代码前必须先读取：

1. `README.md`
2. `开发者规范.md`
3. `src/styles/docs/PIXEL_MALL_STYLE_GUIDE_PROMPT.md`
4. 当前任务相关页面或组件
5. `src/styles/index.css` 和相关样式文件

如果任务涉及后台，必须额外读取：

如果任务涉及数据联动，必须额外读取：

- 购物车 service。
- localStorage 工具。

### 8.2 大模型输出代码前的思考顺序

必须按以下顺序判断：

1. 这个改动对应 README 的哪个功能要求。
2. 是否影响前后台数据联动。
3. 是否需要新增路由。
4. 是否需要新增 service 或复用已有 service。
5. 是否已有可复用组件和 CSS class。
6. 新样式应该放进哪个 `src/styles` 分层文件。
7. 是否需要 loading、empty、error、unauthorized 状态。
8. 是否需要表单校验。
9. 是否需要更新 `Report.md`。
10. 修改后应该运行哪些检查命令。

### 8.3 大模型禁止行为

禁止：

- 不读现有文件就直接重写整个项目。
- 删除用户已有代码或大段样式，除非任务明确要求。
- 把所有页面塞进 `App.jsx`。
- 把 localStorage 操作散落到每个组件。
- 新增一套与 `src/styles` 冲突的 CSS 架构。
- 随机引入大型 UI 库。
- 随机引入状态管理库。
- 为了完成视觉效果牺牲 README 的功能要求。
- 只做静态页面，不实现关键交互。
- 留下无法运行的 import。
- 写出没有权限拦截的后台页面。
- 写出刷新后数据丢失的核心流程。

### 8.4 大模型完成任务后必须检查

必须检查：

## 9. 代码质量标准

### 9.1 可读性

必须做到：

- 变量名表达业务含义。
- 函数不超过合理长度，复杂逻辑拆成小函数。
- JSX 层级清晰。
- 事件处理函数命名清楚。
- 不使用魔法数字表达业务状态。
- 不留下无意义注释。
- 不留下大段注释掉的旧代码。

示例：

```js
const selectedCartItems = cartItems.filter((item) => item.selected);
const totalAmount = selectedCartItems.reduce((sum, item) => {
  return sum + item.price * item.quantity;
}, 0);
```

不要写成：

```js
const a = list.filter((x) => x.s);
const b = a.reduce((x, y) => x + y.p * y.n, 0);
```

### 9.2 错误处理

以下情况必须处理：

- 商品 id 不存在。
- 订单 id 不存在。
- 用户未登录。
- 后台用户无权限。
- localStorage 数据损坏。
- 表单数据不合法。
- 库存不足。
- 商品已下架。

错误处理方式：

- 页面级错误使用明确的 Empty/Error 状态。
- 表单错误显示在字段附近。
- 权限错误显示无权限提示或跳转。
- 数据损坏时使用安全 fallback，不让页面白屏。

### 9.3 可维护性

开发时必须控制文件复杂度：

- 页面文件负责布局和流程。
- 组件文件负责展示和局部交互。
- service 负责数据读写。
- utils 负责纯函数。
- styles 负责视觉。

当一个文件超过约 250 行，应该检查是否需要拆分。但不要为了拆分而拆分。

### 9.4 依赖管理

禁止随意新增依赖。

新增依赖前必须确认：

- README 要求是否真的需要。
- 原生 React、CSS、已有工具是否不能完成。
- 包体积和维护成本是否合理。
- 是否会影响 `npm install` 和打包。

当前不建议新增：

- Redux，除非状态规模明显超出 Context。
- 大型 UI 组件库，除非团队决定整体采用。
- CSS 框架，除非重构整个样式体系。
